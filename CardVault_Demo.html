Raw ciphertext is binary. To store it safely in a database text column, it's Base64-encoded. The IV is prepended to the ciphertext so it's available during decryption. Format: Base64(IV || ciphertext).

Phase 2 — Password Security
Password Hashing
Passwords are never encrypted — they are hashed. Hashing is a one-way process: you can verify a password matches a stored hash, but you cannot reverse the hash to recover the original password.
Encryption vs Hashing — why the difference?
Card numbers need to be retrieved later (for payments), so they're encrypted (reversible with a key). Passwords only need to be verified, never retrieved. Hashing is irreversible — even database admins cannot read a user's actual password.

Why is hashing slow on purpose? The cost factor controls how many rounds of computation happen (2cost iterations). Higher cost = slower. At cost 12 it takes ~300ms — making brute-force attacks take millions of years instead of seconds. The random salt means the same password hashes differently each time.

Phase 2 — Technical Justification
Why These Algorithms?
Every cryptographic decision in this system was made for a specific security reason.

AES-256-GCM Card Fields

What it is: Advanced Encryption Standard, 256-bit key, Galois/Counter Mode.

Why 256-bit? 2²⁵⁶ possible keys — more than atoms in the universe. Brute force is computationally impossible.

Why GCM mode? Unlike CBC, GCM provides both confidentiality (hides data) and integrity (detects tampering). If ciphertext is modified, decryption fails.

Why random IV per field? Encrypting the same card number twice produces different ciphertext, preventing pattern-matching attacks.

Bcrypt Passwords

What it is: A password hashing function designed to be intentionally slow and computationally expensive.

Why not SHA-256? SHA-256 hashes billions of passwords per second. A GPU can crack most passwords in minutes. Bcrypt at cost 12 takes ~300ms — making brute-force 10 billion times slower.

Built-in salting: Random salt prevents rainbow table attacks (pre-computed hash tables).

Why not encrypt passwords? If the key leaks, all encrypted passwords are exposed. Hashing is irreversible.

Key Management

Where is the key stored? Never in the database. In production it lives as an environment variable on the app server, or in a dedicated KMS (AWS KMS / HashiCorp Vault).

Why separate? A stolen database dump only contains ciphertext. Without the key — stored on a separate machine — it's completely useless to an attacker.

Key rotation: Keys are rotated every 90 days. Old records are re-encrypted with the new key.

Application-Layer Encryption

Why not MySQL AES_ENCRYPT()? MySQL's built-in encryption stores the key in the same database as the data — defeating the purpose. If the DB is stolen, so is the key.

Our approach: Encryption happens in the application before data is sent to MySQL. The database only ever receives and stores ciphertext.

The result: Data (DB) and key (app server) are on separate machines. An attacker must compromise both.
